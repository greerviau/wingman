# Review: PR #32 - stop-guard.sh owner-scoped watcher-liveness fix

Reviewer: crew `review-pr-32-github-com-greervia-reviewer`.
Date: 2026-07-12.
Scope: `github.com/greerviau/wingman` PR #32 (`hooks/stop-guard.sh` + `tests/stop-guard.test.sh`), commit `58ac993`.

## Verdict: approve

The fix correctly branches the watcher-liveness pidfile/beatfile pair on `OWNER`, reusing the same `_okey` sanitization the script already derives two lines earlier for `SCRATCH`. Test coverage proves both directions of the original bug, both new tests were confirmed to fail against the unfixed script and pass against the fix, the full test suite is clean apart from one pre-existing, unrelated hang, and the two related liveness checks the PR description claims to have audited check out correct by independent inspection.

## Findings

None that block merge. Two minor observations below, neither a defect.

### 1. [Informational] The fix's correctness rests on `_okey` being computed before use, which it is, but only because of ordering, not a shared function

`hooks/stop-guard.sh:39-44` computes `_okey` inside an `if [ -n "$OWNER" ]` block for `SCRATCH`, and the watcher-liveness block at line 70+ reuses that same variable inside its own, separate `if [ -n "$OWNER" ]` block without recomputing it. This is correct today (both blocks gate on the identical condition, and shell variables leak across `if` blocks in the same script), but it is a slightly fragile pattern: if a future edit ever reordered these blocks or introduced an early `exit` between them, `_okey` could be read unset. Not worth blocking on - the two blocks are five lines apart and the risk is theoretical - but a future refactor that consolidates owner-scoped-filename derivation into one place (e.g. a single `_okey`/`SCRATCH`/`pidfile`/`beatfile` block, or a tiny `okey()` helper) would remove the implicit ordering dependency. No action requested for this PR.

### 2. [Informational] Test isolation is correct but relies on `test_new_home` between the two new cases

The two new tests (`tests/stop-guard.test.sh`, watcher-liveness section) each call `test_new_home` before setting up their fixtures, so the second test's fresh `$WINGMAN_HOME` genuinely has no leftover `watch-lead1.*` files from the first. Confirmed by inspection and by the full suite passing. No finding, just noting it was checked.

## What was verified

- **Branch correctness (primary ask).** Read `hooks/stop-guard.sh` diff directly: when `OWNER` is non-empty, `pidfile`/`beatfile` become `$WM_HOME/watch-$_okey.pid` / `.beat`; when empty (wingman's own top-level layer), they remain the original unscoped `$WM_HOME/watch.pid` / `watch.beat`. This is not a naive "always use owner-scoped" (the `else` branch is present and unchanged from pre-fix) and not a fallback that silently masks the bug (there is no fallback path that tries the global pair when the owner-scoped one is absent - confirmed by the second new test below).
- **`_okey` reuse is the correct pairing.** `bin/watch-fleet` derives its own singleton `PIDFILE`/`BEATFILE`/`WAKEFILE` from `OWNER` (defaulting to `$WINGMAN_CREW_ID`) using the identical sanitization (`tr -c 'A-Za-z0-9._-' '_'`) at `bin/watch-fleet:95-104`, producing `watch-<key>.pid`/`.beat` for a non-empty owner and the legacy unscoped pair otherwise. `stop-guard.sh`'s `_okey` (line 40) uses the same `tr` expression against the same `$OWNER` (`$WINGMAN_CREW_ID`), so the two scripts always agree on the filename for a given owner. Verified by direct comparison of both `tr` expressions, not by assumption.
- **Both directions of the bug are covered by new tests, and both were confirmed to actually test the fix**, not just "something is live":
  - Case 1 (`tests/stop-guard.test.sh`, "a live owner-scoped watcher satisfies the hook"): `WINGMAN_CREW_ID=lead1`, a live process's pid written to `watch-lead1.pid`/`.beat`, **no** global `watch.pid`/`.beat` present. Asserts the hook's output does not mention `watch-fleet` (i.e., does not nudge to arm a watcher). This is the direction that was silently broken pre-fix (the old hardcoded pair would find no global watch.pid and incorrectly nudge despite a healthy owner-scoped watcher).
  - Case 2 (`tests/stop-guard.test.sh`, "a global watcher does not cover an owner with no watcher of its own"): `WINGMAN_CREW_ID=lead2`, a live process's pid written to the **global** `watch.pid`/`.beat`, **no** `watch-lead2.*` present. Asserts the hook's output **does** mention `watch-fleet` (still nudges). This is the case that rules out a fix that just checks "is anything live" - a fix that fell back to the global pair when the owner-scoped one was missing would fail this test, and it does fail on `main` (see below).
- **Both new tests independently reproduce the bug against the unfixed script**, not just pass against the fix. I reverted `hooks/stop-guard.sh` only (kept the new test file) inside a worktree checked out from PR #32 (`git worktree add`, `pr-32` branch) and reran `tests/stop-guard.test.sh`:
  - With the pre-fix script (hardcoded global pair): 14/16 pass, 2 fail - exactly the two new cases. Case 1 fails because the old code only ever checks the global pair, finds no `watch.pid`, and wrongly emits the "no live watcher cycle" nudge despite the live owner-scoped watcher. Case 2 fails with `[] does not contain [watch-fleet]` - the old code finds the *global* `watch.pid` live and incorrectly considers the owner covered, so the hook emits nothing at all where it should nudge.
  - With the fix restored: 16/16 pass.
  - This directly confirms the PR description's own claim of having done this same before/after check.
- **`bin/watch-fleet`'s own `--status` entrypoint** (`bin/watch-fleet:161-166`, `cycle_live()` at `bin/watch-fleet:152-157`): computes `PIDFILE`/`BEATFILE` from `OWNER` at parse time (lines 95-104, same `tr` sanitization as above), before the `--status` case is reached. No bug - it was never susceptible to the class of bug found in `stop-guard.sh` because it has always derived its filenames from `OWNER` rather than hardcoding the global pair. Verified by reading the full owner-scoping and `--status` logic directly, not taken on the PR description's word.
- **`hooks/stop-guard.sh`'s pending-ask liveness check** (`ask/<req>.pid`/`.beat`, lines 100-101): confirmed this is not susceptible to an analogous owner-scoping bug, but for a different reason than "already owner-scoped" - it isn't owner-scoped at all, and doesn't need to be. `$req` is generated by `bin/crew-ask` as `ask-$(uuidgen | ...)` (`bin/crew-ask:254-256`), so request ids are globally unique across every owner on the machine; the pidfile/beatfile pair is already keyed to a single, unambiguous request regardless of which owner is asking or being asked. The outer `pending_asks` list is filtered to the current owner first (`ask-list --from "$OWNER"`, `bin/lib/wm-state.py:1046` `cmd_ask_list`, which matches `record.get("from") == args.sender`), so there is no scenario where one owner's hook reads another owner's ask liveness file by an id collision. The developer's conclusion in the PR description ("scoped per-request, not per-owner, so it has no analogous issue") is correct.
- **No regression to non-lead (top-level wingman) behavior.** The `OWNER` unset branch is byte-for-byte the pre-fix logic (`pidfile="$WM_HOME/watch.pid"`, `beatfile="$WM_HOME/watch.beat"`), and all 14 pre-existing tests in `tests/stop-guard.test.sh` (none of which set `WINGMAN_CREW_ID`) pass unchanged on the fixed script.
- **Full suite regression check.** `bash tests/run.sh` against the PR #32 worktree stalls on `tests/crew-resume.test.sh`. Verified independently that this same hang occurs on unmodified `main` (ran `tests/crew-resume.test.sh` directly against `main` with a 60s timeout; killed identically), confirming the PR description's claim that this is pre-existing and unrelated to this change, not a regression it introduces. With that one suite excluded, every other suite in `tests/*.test.sh` (20 files) was run individually to completion against the PR #32 worktree and all pass cleanly: `ack-dedup` (18/18), `crew-ask-guardrail` (10/10), `crew-ask` (23/23), `crew-say-guardrail` (9/9), `dead-lead-orphans` (21/21), `group-attention` (19/19), `handled-marker` (27/27), `no-direct-edit-guard` (54/54), `owner-scope` (21/21), `playbook-resolution` (16/16), `pr-eval` (9/9), `pr-watch` (11/11), `review-state` (10/10), `roster-cleanup` (20/20), `spawn-scope` (19/19), `stall-check` (31/31), `stop-guard` (16/16), `submit-delivery` (3/3), `watch-fleet` (72/72). No regressions anywhere in the suite from this change.
