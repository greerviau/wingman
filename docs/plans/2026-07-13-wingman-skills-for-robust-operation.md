**r4** (this revision) - A third-pass review (`docs/analysis/2026-07-13-third-review-wingman-skills-for-robust-operation-plan.md`) confirmed the stdin/guard fix (r3 finding 1) fully closed by running the guard's real resolver against both the r2 and r3 shapes, and confirmed the `remote-control-dropped` should-fix structurally closed. 1. **(Must-fix.)** The 3-within-60-seconds budget is a *rate* detector whose trip condition depends on wingman's own model-turn latency between classify calls - an unbounded quantity the design never measured. Worked through algebraically: with log entries spaced by the full loop period `P` (death time plus wingman's wake-classify-rearm latency), the third entry falls inside a 60s window only when `2P <= 60`, i.e. the budget trips **only if `P <= 30s`**. Any persistent failure recurring slower than that - a busy wingman turn, or a death that itself takes 20-30s - never trips it and reproduces the exact silent livelock the fix was meant to close. The plan's own regression test for this case drove `P` toward zero (rapid-fire `SIGKILL`s), which is precisely the one regime where the defective design passes.
2. **(Must-fix.)** `--classify`'s two hint branches (`sigkill-suspected` / `clean-exit-or-sigterm`) are not exhaustive - the omitted third state, "pidfile present and its pid alive," is reachable through the exact redundant-arm race `CLAUDE.md` and the plan's own F2 fix both describe (two live `watch-fleet` processes for one owner, one owning the record, the other about to). When the interleaving lands wrong, a perfectly healthy cycle gets classified `spurious`, pollutes the forensic log, and feeds the (now also-fixed) failure budget toward a false `spurious-repeated` that stops supervision while a healthy cycle is running the whole time.
3. **(Should-fix.)** The plan never states where `--classify` is dispatched relative to the claim-lock acquisition. Falling through to the arm path by omission would be silently severe: with a cycle live, a stray `--classify` invocation would hit the singleton guard and, under this very plan's own design, write `healthy` to the exit record and exit - meaning every classify call would return `healthy` regardless of what actually happened, restoring F1's original swallow-every-event failure through a new door.
4. **(Nit.)** The `spurious-repeated` pilot-facing message names the log to read but not how supervision resumes.

This revision replaces the wall-clock window budget with a consecutive-count budget (immune to `P` by construction) qualified by how long the dead cycle survived, so an isolated long-lived reap is not conflated with "cannot stay up"; adds an explicit `cycle_live` check to `--classify` so a redundant-arm race is classified `healthy`, never `spurious`; states the exact dispatch point (`--classify` joins the existing `--status`/`--stop` mode block, before claim-lock acquisition, and never touches `$PIDFILE`/`$BEATFILE`); and gives the `spurious-repeated` message its own recovery instruction.

2. **(Must-fix, the structural one.)** `watch-started` is written only on a **successful claim**. The plan's own named reproduction (`watch-fleet:290`'s stuck claim-lock death) dies *before* ever claiming, so it never writes `watch-started` at all, and every one of the plan's own lifetime rules then routes that case to "log only, do not count." 3. **(Should-fix.)** The hint logic's exhaustiveness claim was false - a fourth, reachable state (pidfile present, pid alive, beacon stale beyond `GRACE`) matches neither branch, and under `set -u` an implementation with no terminal `else` aborts `--classify` outright.

5. **(Should-fix.)** The claim-time stale-record clear, introduced as "cheap" defense-in-depth in r3, is not cheap: because `--classify` deletes every record it successfully reads, the only way a record survives to a fresh claim is that a wake was never classified - i.e. exactly the case where clearing it destroys the last trace of an already-acked event.
6. **Three nits:** `bin/watch-fleet`'s own argument parser (`watch-fleet:82-90`) does not yet accept `--classify` and would `wm_die` on it before ever reaching the dispatch block the plan cites; "never touch `$PIDFILE`/`$BEATFILE`" should say "never *mutate*" since `cycle_live` legitimately reads both; and the two new owner-keyed state files have no stated cleanup lifecycle.

The reviewer's own framing for finding 2 is decisive: *"consider whether the rate-based budget approach itself is the wrong mechanism for this case... rather than patching the threshold or the qualifier again."* Three rounds of increasingly narrow timing-based fixes to the same underlying coupling (a wall-clock measurement taken inside a wingman turn, whose latency the design does not and cannot control) is the pattern this spec's own operating principles weigh against - prefer correctness, simplicity, and robustness over the engineering cost of one more patch. This revision takes the reviewer's own offered alternative: **the lifetime qualifier is dropped entirely.** The budget becomes a pure consecutive count - incremented on every genuinely spurious classification, reset on any non-spurious one, no wall-clock term anywhere in the trip condition. This closes findings 1 and 2 at the root (there is no `now`, no `watch-started`, and therefore no coupling to `W` or a gap for "died before claiming" to fall through), at the cost the reviewer names explicitly and accepts: a run of purely coincidental, long-lived, unrelated reaps with no intervening event between them would now count toward the same budget a genuinely persistent failure would - rare, and the resulting message ("the watcher has died three times in a row with no successful cycle in between") is true in that case too, so the false-positive cost is far smaller than the false-negative it replaces. The should-fixes and nits are addressed in the same revision - see the relevant subsections below.

## Objective

Wingman already wraps its core crew verbs - `spawn`, `status`, `blocked`, `lead`, `standdown`, `prune`, `takeover` - as project slash-command skills under `.claude/commands/`.
This spec assesses three candidate additions - `watch`, `say`/`ask`, and `prefs` - against the same convention, and recommends a priority order grounded in an observed failure: in a live session, the orchestrator forgot to re-arm the watcher once (causing a stuck stop-hook loop) and separately saw `watch-fleet`'s background process get killed externally twice, each requiring ad hoc manual recovery.

## The existing convention

Every current skill (`.claude/commands/*.md`) is a small markdown file with three parts:

- **Frontmatter**: `description` (one line, shown in `bin/spawn-crew --list-types`-style listings), `argument-hint` (the expected `$ARGUMENTS` shape), and `allowed-tools` - a narrow `Bash(bin/<script>:*)` pattern (sometimes two, e.g. `spawn.md` allows both `bin/spawn-crew` and `bin/discover-projects`) that pre-authorizes exactly the command(s) the skill needs, so invoking it never triggers an interactive permission prompt.
- **A short prose body** that tells the session how to parse `$ARGUMENTS`, which `bin/` command to run, and how to report the result back to the pilot. There is no branching state machine in any of the seven existing skills - each is close to a 1:1 wrapper around one `bin/` verb, with light argument parsing and a fixed reporting shape.
- **No skill spans a wake boundary.** All seven complete in the turn they're invoked. None of them arm a background task or expect to be re-entered later.

`/watch` breaks this last property by necessity (it has to span a wake), which is exactly why - per the review - its branching logic does not belong in skill prose the way the other six do. The fix below keeps the skill itself a thin wrapper and moves the actual state machine into `bin/`, restoring the convention rather than being the one skill that quietly violates it.

Checked and confirmed by direct inspection: `bin/crew-say` and `bin/crew-ask` appear in **no** `allowed-tools` list and in neither `.claude/settings.json` nor `.claude/settings.local.json`'s permission allowlists. `blocked.md`'s own body instructs "relay my answer down with `bin/crew-say`" but its `allowed-tools` covers only `bin/crew-list` - so today, answering a blocked crew member's decision triggers an interactive permission prompt every time, unlike every other core verb. This is a concrete, verified gap, not a hypothetical one, and it is independent of everything else in this spec - the review confirmed it and found no issues with the `/say`/`/ask` design.

## Investigation: why does `watch-fleet`'s background process get killed unexpectedly?

This was investigated empirically and against the codebase's own history, not just theorized. Findings, in the order they were ruled in or out:

### Ruled out: a harness-level Bash `run_in_background` timeout, up to 300s

The Bash tool's own description states a default 120000ms timeout and a 600000ms (10 min) cap, without explicitly exempting backgrounded commands. Since `watch-fleet` blocks silently (no stdout) between polls, potentially for far longer than 10 minutes during a genuinely quiet period, this looked like a plausible structural mismatch worth testing directly rather than assuming.

**Test performed:** armed `sleep 300 && echo SLEEP_DONE_300` via `Bash` with `run_in_background: true` and no explicit `timeout` parameter. It ran to completion (exit 0, full output) at the full 300 seconds, uninterrupted - past the documented 120s default. This rules out the naive "the default timeout kills background tasks" theory, at least up to 300s.

### The 600s-cap question (r1 left this open; r2 resolves it - see F5 below)

r1 flagged, but did not test, whether the 600000ms (10 min) cap applies to backgrounded tasks and recommended setting `timeout: 600000` explicitly as what it called a "free" defensive floor. The review (F5) correctly identified this as unverified and risky in either direction (a no-op if the parameter is ignored for background tasks, or a new 10-minute self-kill cadence if it is honored). r2 ran the missing test; see "Resolution of F5" under the `/watch` design below for the result and the resulting recommendation.

### Confirmed precedent: a live watcher has been killed by hand before, and the fix was wording-only

`bin/watch-fleet`'s own singleton-guard report (`watch-fleet:302`) reads: *"watcher: already armed and healthy... this pid is the EXISTING watcher, never a target to stop or kill."* Git history (commit `bfb1df8`, closing issue #12) shows this exact phrasing was added specifically because **a live, healthy watcher had previously been killed by hand** after its printed pid was misread as an instruction to act on. `CLAUDE.md` independently states the same rule in prose: *"Never `kill` a watch-fleet process for any reason during normal operation - the pid shown in a `healthy`/`armed` line is informational, never an instruction."*

Both fixes are **wording only**. Nothing mechanically prevents `kill <pid>` / `pkill` / an equivalent from being run against a live `watch-fleet` pid, unlike other safety-critical rules in this codebase (`hooks/no-merge-guard.sh` for PR merges, `hooks/pilot-preferences-guard.sh` for onboarding preferences), which are backed by a `PreToolUse` hook that denies the dangerous shape outright rather than relying on a session remembering not to do it.

### Related but distinct: the systemd autostart cgroup-reap bug (issue #45, closed)

This box runs wingman under a systemd `--user` timer (`wingman.timer` → `wingman-ensure.service` every 5 minutes for self-heal, `wingman.service` for explicit restart). Issue #45 (closed) documented that these units' default `Type=oneshot` + `KillMode=control-group` reaped every process left in the service's cgroup the instant `ExecStart` exited - including a freshly-forked tmux server and everything hosted inside it. The fix (`wm_tmux_may_fork_server`, wrapping any server-forking tmux call in a transient `systemd-run --user --scope --collect`) is live in the current `start-wingman.sh`, dated today.

This is real, dated, and directly relevant history - it proves this exact box has already produced a silent external kill of exactly this shape once. It is presented as related context rather than the confirmed mechanism for the specific symptom described, because a full cgroup reap would be expected to take down the *entire* wingman session, not selectively kill only its backgrounded child while the parent keeps running and is able to notice and manually recover.

### A hypothesis the review correctly added: harness-side reaping across `/clear` or context compaction

The review's F9 identified a gap in the original hypothesis set that fits the observed shape (child dies, parent survives and can react) better than either mechanism above: a harness-side reap of tracked background tasks across a `/clear` or a compaction event would kill only the backgrounded child, leave the orchestrating session alive, and would recur on an ordinary cadence given how frequently a long-running orchestrator session compacts. This was not tested (it requires triggering a compaction event under controlled observation, which was judged out of scope for this spec-level investigation) and is recorded here as an open hypothesis rather than a confirmed mechanism - see "What remains open" below.

### The self-kill guard is a prerequisite, not a substitute, and does not cover every mechanism

Filed **[issue #64](https://github.com/greerviau/wingman/issues/64)**: add a `PreToolUse` guard hook, mirroring `hooks/no-merge-guard.sh`'s shape, that denies a `kill`/`pkill`/`tmux kill-*` command whose target resolves to a pid currently recorded as a live `watch-fleet` pidfile. This closes the self-inflicted-kill class structurally.

The review correctly noted (F9) that this guard can only deny a kill **issued as a tool call by the session it guards** - it does nothing against a harness-side reap across compaction, a systemd-level event, an OOM kill, or a stray signal (this codebase already carries scar tissue for exactly that last case: `watch-fleet:73`'s unconditional `trap '' URG`, added because a stray SIGURG once terminated the loop with exit 144). Issue #64 is corrected here from r1's framing ("a prerequisite for `/watch`'s design to be complete") to: **a necessary fix for the one confirmed, precedented mechanism, and a complement to - not a substitute for - `/watch`'s own graceful recovery from *any* cause of unexpected death**, since #64 provably cannot cover every mechanism in the hypothesis set.

### What remains open

The forensic capture this spec proposes (see the `/watch` design's `--classify` verb below) is the concrete answer to "what remains open": a SIGKILL leaves `$WM_HOME/watch.pid` behind (the `INT TERM` trap at `watch-fleet:311` cannot run for `SIGKILL`, which is never trappable by POSIX design), while a clean exit or a `SIGTERM` removes it. Recording that distinction every time a spurious death is observed is what would eventually turn "one of four hypotheses" into a confirmed root cause, without needing a dedicated investigation up front.

### The stuck stop-hook loop is a separate, not-yet-root-caused bug

r1 cited the stuck-stop-hook-loop incident as its headline justification for `/watch` without explaining why the loop got stuck - the review (F8) correctly flagged this as a gap, since `hooks/stop-guard.sh` is explicitly designed not to loop (it respects `stop_hook_active`, blocks once per pass, and pass 2 marks the captured scratch set handled unconditionally). ### 2. `/watch` - build second, after the `bin/` rework below

**Worth building? Yes, once reworked as specified here - not as originally proposed.** r1's design would have shipped a skill that silently swallowed every genuine crew event (F1) and could livelock on a redundant arm (F2). Both are fixed below by moving the exit classification into a new, testable `bin/` verb instead of skill prose (F3), which is also what the existing-convention section above argues is the right shape for any skill with real branching logic.

**What a skill can and cannot fix here, stated precisely:** a slash command is not a callback - the harness still re-invokes the idle session directly with the background task's stdout as the wake signal, exactly as it does today; no skill changes that mechanism. What a skill changes is that the *procedure* run in response to a wake stops being four scattered bullet points in `CLAUDE.md`'s "The wake loop" section and becomes one named, fixed command - but only if that command's own logic is correct and tested, which is the entire content of this rework.

#### F1 fix: classify by scanning for a reason prefix anywhere in the block, not the first line

`bin/watch-fleet` always prints its own arm-time line to **stdout first** (`watch-fleet:312`, `wm_ok "watcher: armed pid=$$ (interval ${INTERVAL}s)"` - confirmed `wm_ok` writes to stdout, not stderr, unlike `wm_warn`/`wm_err`). So a genuine fire's full captured output looks like:

```
✓ watcher: armed pid=12345 (interval 5s)
review: <crew-id> <note>
--
The lines above are state-change deltas, not the full picture.
...
```

No real fire's stdout *begins* with a reason prefix; the fix is to scan every line up to the literal `--` separator (`watch-fleet:424`) for a match against `^(blocked|review|done|died|stalled|remote-control-dropped):`, not just the first line. This prefix list is confirmed complete against the current code: `wm_state group-attention` (`wm-state.py:934,945`) keeps its synthetic correlated rows (mass-death, correlated API outage) under the existing `died`/`stalled` statuses, so no additional prefix is introduced by batch grouping.

#### F2 fix: a third classification outcome, `healthy`, means "do nothing," never "spurious"

A redundant arm against an already-live cycle prints exactly one line and exits immediately (`watch-fleet:301-305`):

```
✓ watcher: already armed and healthy - one cycle is live (pid N, beacon Ms ago). Nothing to do; this pid is the EXISTING watcher, never a target to stop or kill.
```

Because a harness-tracked background task's completion is itself a wake, this near-instant exit produces a near-instant notification. r1's classifier had no branch for this and would have logged it as "spurious" and re-armed - hitting the same live cycle again, producing the same instant exit and notification again, in a self-sustaining loop. This is corrected as the review requires: **three** classification outcomes, not two, and `healthy` means the turn ends with no report, no log write, and no re-arm, because a live cycle already covers supervision. This also restores (rather than silently reverses, as r1 did) `CLAUDE.md`'s existing instruction to read `healthy` as "do not churn extra arms."

#### F3 fix (r3): the classifier moves into `bin/watch-fleet --classify`, reading a `watch-fleet`-written exit record - not stdin

r2 specified `--classify` as reading the just-completed cycle's captured stdout on stdin, and never named the exact shell command that would deliver that text to it. The r3 re-review ran `hooks/pilot-preferences-guard.sh`'s own `command_segments` resolver (`hooks/lib/cmd_match.py:53`, which splits on `;`/`&&`/`||`/`|` **and newlines**) against every plausible shape and found all of them denied: a `printf ... | bin/watch-fleet --classify` pipe fails because `printf` is not an allowlisted basename; a heredoc fails because the guard re-lexes the heredoc's own captured prose line-by-line, and lines like `review: <id> <note>` or `The lines above are state-change deltas...` don't resolve to anything allowlisted either. This is exactly the class of defect the prior review's F10 caught for `/prefs` (a grant that cannot match the command actually invoked) - reintroduced here in a different shape. It also broke the F4 `/watch`-half fix: the gate-time exemption covers a *bare* `bin/watch-fleet` invocation (arming), never a piped or heredoc'd one (processing), so the "processing falls back to the raw command too" claim in r2 was never true.

**Fix: drop stdin. Have `bin/watch-fleet` record its own exit reason to a small file at each exit point, and have `--classify` read that file with no stdin at all.**

New state, owner-keyed exactly like `$PIDFILE`/`$WAKEFILE`: `$WINGMAN_HOME/watch-exit-<owner-key>` (bare `watch-exit` for owner `""`), plus `$WINGMAN_HOME/watch-spurious-count-<owner-key>` (r5's consecutive-count budget - see the finding-2 fix below; r4's separate `watch-started-<owner-key>` file is **removed in r5**, not merely unused - see that fix for why). Places in `bin/watch-fleet` that change:

1. **`bin/watch-fleet`'s own argument parser** (`watch-fleet:82-90`, the `case "$1" in --owner) ...; --status|--stop|--start|arm|__loop) MODE="$1"; shift ;; ...` block) **must add `--classify` to the set of tokens that set `$MODE`** (r5 nit fix: the plan previously named only the later `case "$MODE"` dispatch block and never this earlier parser, which is the one place an unrecognized `--classify` would actually die today, via the existing `*) wm_die "unknown arg: $1"` catch-all).
2. **At claim time** (right after `echo $$ > "$PIDFILE"`, `watch-fleet:308`, when a genuinely fresh cycle is won): **before** clearing any stale exit-record, if one exists, append it to `$WINGMAN_HOME/watch-spurious.log` as a `dropped-wake` forensic line (owner, the record's contents, timestamp) - **then** clear it (r5 finding-5 fix: r3's "cheap" silent clear could permanently destroy the last trace of an already-acked `fire` that was never classified - see that fix below for the full reasoning). r4's `watch-started` write at this point is removed along with the file itself.
3. **The singleton `healthy` early-exit** (`watch-fleet:301-305`, before `release_claim; exit 0`): write `healthy` to the exit-record for this owner, **but only if no record is currently present** (r5 finding-4 fix: this write must be non-clobbering - see below for why).
4. **`fire()`** (`watch-fleet:399-437`, before its existing `rm -f "$PIDFILE"; exit 0`): write `fire` to the exit-record **unconditionally** (always overwrites - this is the highest-priority writer; see finding 4).
5. **The `remote-control-dropped` path** (`watch-fleet:451-460`, before its existing `rm -f "$PIDFILE"; exit 0`): write `remote-control-dropped` to the exit-record **unconditionally**. This is also the r3 fix for finding 3 - this path gets its own explicit, correct record rather than needing to be inferred from stdout text at all.

Any exit that is **not** one of these three writer points - a signal, `wm_die` (e.g. `watch-fleet:290`'s stuck-claim-lock death), a harness-level kill - writes nothing, so the exit-record simply stays absent. Absence is *necessary but not sufficient* for "spurious" - see the finding-2 `cycle_live` check below for why a second check is required before drawing that conclusion.

**`--classify`'s dispatch point:** `--classify` is a `case "$MODE"` branch alongside the existing `--status`/`--stop`/`--start` handling (`watch-fleet:225-245`), and like those it **returns before the claim-lock acquisition** (`watch-fleet:271` onward) - it must never call `mkdir "$CLAIMLOCK"`, never *mutate* `$PIDFILE`/`$BEATFILE` (it legitimately *reads* both, via the `cycle_live` check below - r5 wording fix, since "never touch" was imprecise), and never enter the blocking loop. Falling through to the arm path instead would be silently severe: with a cycle live, a stray `--classify` invocation would hit the singleton guard and - under this plan's own design - write `healthy` to the exit record and exit, meaning *every* classify call would report `healthy` regardless of what actually happened, restoring F1's original swallow-every-event failure through a different door, plus a 5-second hang per call whenever the claim lock happens to be contended.

`bin/watch-fleet --classify [--owner <id>]` (dispatched as above, no stdin):

1. If the exit-record file exists: read its one-word contents, print it (`healthy`, `fire`, or `remote-control-dropped`), delete the record (consumed - the next classify call must not see it again), exit 0.
2. **If absent, check `cycle_live` (`watch-fleet:218-223`, the same helper the singleton guard itself uses) before concluding anything:**
   - **A cycle is live** → print `healthy`, exit 0. No log write, no budget tick, no re-arm needed. This is the state r3 omitted: the exit-record is a single owner-keyed slot, but two `watch-fleet` processes can legitimately be in flight for one owner at once (a redundant arm racing a live cycle is exactly what the `healthy` outcome exists to describe, per `CLAUDE.md` and the F2 fix above) - so record-absent does not, by itself, mean nothing is running. Checking `cycle_live` first closes this by construction for every "more wakes than record slots" interleaving on the *read* side; the *write* side of the same class of race (finding 4) is closed separately, below.
   - **No cycle is live** → genuinely spurious. Determine the forensic hint. **Checked first, before anything else (r6 addition, the fifth-pass review's should-fix): `[ -d "$PIDFILE.lock" ]`** → `stale-claim-lock`. This is the one hint that is actually about the arm whose death is being classified, rather than about pidfile residue left behind by an entirely different, earlier cycle - and it is the correct diagnosis for the plan's own headline repro (`watch-fleet:290`'s stuck claim lock), where the other three hints below would otherwise report `sigkill-suspected`/`clean-exit-or-sigterm` against the *original* `SIGKILL`ed cycle and never mention the lock actually blocking every subsequent arm. If no lock directory is present, fall through to an **explicit, exhaustive three-way check written as `if`/`elif`/`else`** (r6 wording fix: the branch below was previously phrased as a third *condition*, which happens to be total today only because `beat_age()` returns `999999` when `$BEATFILE` is missing - an accident of an unrelated helper, not a property this design should lean on under `set -u`; write it as the unconditional `else` arm instead):
     - `$PIDFILE` absent → `clean-exit-or-sigterm`.
     - `$PIDFILE` present, its pid no longer alive → `sigkill-suspected` (a `SIGKILL` cannot run the `INT TERM` trap at `watch-fleet:311`, so a survived pidfile pointing at a dead pid is diagnostic).
     - else (pidfile present, pid alive - `cycle_live` already ruled out "and beacon fresh," so this arm is left with "and beacon stale") → `hung-or-stale-pidfile` (a wedged process - a hung subprocess, a `SIGSTOP`, or a recycled pid - not a healthy cycle and not a clean death either). This is the fourth, previously-unhandled state the third-pass review found reachable and mis-scored by omission.

     The lock check plus these three branches are now exhaustive over everything that reaches this point (`cycle_live` false, by the check above), with no implicit gap. Append one line to `$WINGMAN_HOME/watch-spurious.log` (timestamp, owner, hint), then apply the failure budget - see the finding-2 fix immediately below.

   **`--classify`'s stdout for the two spurious outcomes carries the hint on the same line as the outcome word (r7 addition - see the finding-3 fix below for why), plus the trip count itself (r8 addition - see the r8 revision-history entry above): `spurious <count> <hint>` / `spurious-repeated <count> <hint>`, field order fixed (count first, then hint), e.g. `spurious-repeated 3 stale-claim-lock`.** A uniform two-field shape for both spurious outcomes, so the parse rule is one sentence regardless of which of the two was received. `healthy`/`fire`/`remote-control-dropped` remain bare single-token lines, since neither field applies to them.

The command the skill actually runs is a single bare `bin/watch-fleet --classify` (plus `--owner <id>` when relevant) - no pipe, no heredoc, no stdin round-trip of ANSI-coloured multi-line text through shell quoting. This passes the skill's own `Bash(bin/watch-fleet:*)` grant by construction (it is a `bin/watch-fleet` invocation, full stop) and, as a direct consequence, resolves the `/watch`-half of F4 too: during the pending-preferences window, *both* arming and classifying are now bare `watch-fleet` invocations covered by the guard's existing exemption for that basename - the r2 caveat (run the raw command directly rather than through `/watch` during the gate-pending window, since invoking the `Skill` tool itself is what the guard denies, independent of what commands the skill's body would go on to run) still applies and is now sufficient, with no remaining gap.

Every subcommand branch, the record-write at each exit point, and the dispatch-point boundary itself are unit-testable in `tests/` the same way every other `bin/` verb in this codebase is - closing the "untestable prose, guaranteed to drift" risk the original review raised, and doing so more robustly than r2's text-scanning design: `watch-fleet` now tells `--classify` exactly what happened via a value it wrote itself, rather than `--classify` re-deriving that fact by pattern-matching text that was designed for a human to read.

#### Finding 2 fix (r5, corrected in r6): drop the lifetime qualifier entirely - a pure consecutive count, no wall-clock term anywhere, with a stated writer invariant and reset-on-trip

**Why r4's fix was still broken, in a new form.** r4 claimed the consecutive-count budget was "immune to wingman's turn latency by construction." That was true of the *counter* and false of the *lifetime qualifier gating every increment of it* (`now - watch-started`, computed inside `--classify`, which runs inside a wingman turn - so `now` is death time plus `W`, wingman's own wake/model-turn latency, exactly the unbounded quantity the third pass rejected r3 for depending on). Worked through precisely: the resulting trip condition (`L + W < 15s`, the default `WM_SPURIOUS_MIN_LIFETIME`) is **strictly stricter** than r3's disproven one (`L + W <= 30s`) - a failure r3's broken budget would have caught (a 5s death behind a 20s wingman turn) is silently missed by r4's.

Worse, and structural rather than a threshold problem: `watch-started` is written only on a **successful claim** (per the F3 fix above). The plan's own named reproduction - `watch-fleet:290`'s stuck claim-lock death - dies *before* ever reaching that point, so it never writes `watch-started` at all. Every one of r4's own lifetime rules then routes that exact case to "log only, do not count": the file is either missing (never claimed) or stale (an earlier cycle's timestamp, read as an implausibly long lifetime). Traced against the repro: a kill landing inside the sub-second claim window (`watch-fleet:278-303`) leaks the claim-lock directory - a *blocking* cycle has already released the lock before it ever enters its loop, so only a kill landing in that narrow window leaks it, which a *redundant arm* (spending most of its short life inside exactly that window) hits far more readily than a long-lived blocking cycle ever would. Every subsequent arm spins the full 50 tries and dies at the same `wm_die`, without claiming, without writing an exit-record, without writing `watch-started`; `--classify` sees record-absent, `cycle_live` false, hint `sigkill-suspected`, lifetime computed against a stamp that only grows staler - "log only, do not count," forever. The count never leaves zero. `spurious-repeated` never fires. This is the third pass's finding 2, verbatim, surviving three consecutive attempts to close it through the same mechanism (a rate/lifetime measurement taken at classify time).

**Fix: stop measuring anything at classify time. Take the pure consecutive count the third-pass review itself offered as its "minimum sufficient" alternative.**

**Invariant (r6 addition, stated explicitly because its violation was the fifth-pass review's must-fix 1): the count file has exactly three mutation rules, and all three live inside `--classify`. Nothing else in `bin/watch-fleet` - in particular, no code at claim time - ever writes to it.** A claim-time reset was drafted into an earlier version of this plan's "Files touched" list without ever being stated here, contradicting this section; it is deleted, not merely unmentioned, because it silently reverts the whole fix for any failure that claims successfully before dying (the originally-observed external-kill case) while leaving the plan's own never-claims repro passing - the exact "test exercises only the regime where the defect is invisible" trap the third and fourth passes each caught in a different shape. Winning a claim proves only that the lock was free, not that the cycle can stay up, so it is not a valid reset signal on its own terms even setting the regression aside.

- **State:** only `$WINGMAN_HOME/watch-spurious-count-<owner-key>` (a plain integer). No lifetime file, no wall-clock term anywhere in the trip condition. **A missing or unreadable count file reads as 0 (r8 addition):** `bin/watch-fleet` runs under `set -u`, so this must hold without ever reading an unassigned variable. Reading the file via command substitution never errors on its own when the file is absent (`cat` fails silently into an empty string); the implementation then normalizes both "empty" and "anything non-numeric" to `0` *before* the arithmetic increment runs, so `$((count + 1))` always sees an assigned value. The alternative shape - assigning straight into an arithmetic context without that normalization - aborts under `set -u` on the very first classify call of a fresh owner, when the count file has never been written.
- **Rule 1 - increment.** On every genuinely spurious classification (i.e., after the `cycle_live` check has already ruled out a live cycle - no exception for "died before claiming"; that case is now counted exactly like any other spurious death, which is the entire point): increment the count file, unconditionally.
- **Rule 2 - reset on health.** On any non-spurious classify *outcome* (`healthy`, `fire`, `remote-control-dropped`): reset the count file to 0. This is *necessary* for self-healing but, on its own, **not sufficient** - see the r6 correction immediately below.
- **Rule 3 - reset on trip (r6 addition, the fifth-pass review's must-fix 2).** When the incremented count reaches `WM_SPURIOUS_BUDGET_COUNT` (env-overridable, default 3) and `--classify` prints `spurious-repeated`, it also resets the count file to 0 in the same call - **the trip consumes the budget.**

If the incremented count reaches the threshold, print `spurious-repeated` (with the hint appended on the same line, per the F3 fix above) instead of `spurious`, also resetting as part of the same call (rule 3); otherwise print plain `spurious` (also hint-appended).

**Why rule 3 is required, not optional, and why rule 2 alone was a false claim of self-healing.** An earlier draft of this fix relied on rule 2 alone and described it as making the budget "self-healing once a cycle actually stays up again." That is false on a quiet fleet: rule 2 fires only on a *classification*, and a classification only happens when a wake is processed - a genuinely healthy, blocking cycle produces **no wake at all**. On an idle or parked fleet (crew sitting in `review`, nothing transitioning - normal and expected, per this spec's own cost-discipline guidance), a healthy cycle can run for days without ever resetting the count, because there is nothing for it to reset *from* a classification's perspective. Composed with a trip that never resets (the earlier draft's rule), the result is sticky and permanently wrong: after any single trip, the count sits at or above threshold indefinitely (a manual re-arm following the plan's own recovery message doesn't classify anything either, since it skips `/watch`'s step 1), and the *next* isolated, unrelated spurious death - the ordinary case this budget exists to absorb silently - immediately re-trips `spurious-repeated` with a message that is now false ("died three times in a row" when only one death occurred since the last genuine recovery) and a refusal to re-arm that is now wrong (a silent re-arm was the correct response). Rule 3 closes this: the trip itself is the signal that gets consumed, so the very next classification after a trip - spurious or not - starts from a clean slate, and a single stray death weeks later is once again absorbed as ordinary `spurious`, not an immediate false re-trip.

**The accepted tradeoff, stated plainly rather than assumed away.** Dropping the lifetime qualifier reopens the false positive it existed to prevent: three genuinely consecutive, unrelated, long-lived reaps with no successful cycle in between would still trip `spurious-repeated`. This is the exact tradeoff the third-pass review named when it offered this option, and its own assessment is adopted here rather than re-litigated: that scenario is rare, and the message it produces is true in that case too, so the cost of an occasional over-eager surfacing is far smaller than the cost of the silent, permanent supervision loss this qualifier was actually causing across three revisions. Rule 3 narrows this further rather than widening it: because a trip now resets the count, only three *quick, unrepaired* consecutive failures ever trip it, never a lone stray death long after a genuine recovery. A codebase-consistent, lower-effort way to widen the true-positive/false-positive tradeoff later, if the false-positive rate proves material in practice, is a longer-window variant of the same pure count - deliberately not pursued here, since it reintroduces exactly the kind of subtlety five rounds of review have now spent finding bugs in, for a problem not yet observed to exist.

`/watch`'s body branches on the result: `spurious` behaves as before (report nothing, arm one fresh cycle, end the turn). `spurious-repeated` does the opposite - **do not re-arm.** Report to the pilot instead, with a remedy that depends on the forensic hint (see the finding-3 fix below for why a generic remedy is not always correct): *"the watcher for `<owner>` has died `<count>` times in a row with no successful cycle in between (see `$WINGMAN_HOME/watch-spurious.log`); fleet supervision is not being maintained. <hint-specific remedy>."* `<count>` is the real value `--classify` just printed as the second field of this outcome line (the field right after the outcome word itself; r8 fix - see the r8 revision-history entry above for why interpolating `WM_SPURIOUS_BUDGET_COUNT` or any other guessed value was wrong), not a value `/watch` infers or assumes. This turns a watcher that cannot stay up into the genuine attention event it actually is, matches how every other unrecoverable condition in this codebase is handled (surfaced, not silently absorbed), and gives `watch-spurious.log` a concrete trigger for actually being read rather than accumulating unread forever.

#### Finding 3 fix (r6): the hint must name the current arm's own cause of death, not just the prior cycle's pidfile residue - and the recovery message must match the hint

The pidfile-residue hint logic (F3 fix, above) answers "what happened to the *cycle that was running before*," which is the right question for a `SIGKILL`/`SIGTERM`/reap of a cycle that had successfully claimed. It answers the wrong question for the plan's own headline repro: a stuck claim lock kills every *subsequent arm attempt* before any of them ever claims, so there is no "cycle that was running before" for this specific death - the pidfile residue being inspected belongs to whatever much earlier cycle originally leaked the lock, not to the arm that just died at `watch-fleet:290`. The result, traced against the repro: `--classify` reports `sigkill-suspected` (a true statement about the original, long-gone cycle) three times in a row, `spurious-repeated` fires, the pilot goes to `watch-spurious.log`, and every line points them at "something killed the watcher" - never at the leaked `~/.wingman/watch.pid.lock` directory actually causing every failure since. The plan's own generic recovery instruction ("run `/watch` again or arm `bin/watch-fleet` directly") then makes things worse by implying a fix that cannot work: the lock is still there, so the next arm dies identically.

**Fix:** the `stale-claim-lock` hint (added to the F3 fix's hint logic above, checked only after `cycle_live` and only within the "no cycle live" branch - never ahead of it, since a leaked lock coexisting with a genuinely healthy, live cycle must still classify `healthy`, not `spurious`) is the detection mechanism; this is its consequence for the pilot-facing message. `spurious-repeated`'s report is no longer one fixed sentence - the remedy clause is conditional on the hint.

**How `/watch` actually obtains the hint (r7 correction - see the should-fix below for why this needed fixing again): `--classify` prints it directly on its own outcome line** (`spurious-repeated 3 stale-claim-lock`, per the F3 fix above and the r8 count-field addition), which `/watch` already has in hand from the very `Bash` call it just ran - no separate read of `watch-spurious.log` or anything else, and therefore no additional tool grant. `/watch`'s body parses a `spurious`/`spurious-repeated` outcome line as outcome word, then count, then hint, and selects the remedy from the hint directly:

- Hint is `stale-claim-lock`: *"...fleet supervision is not being maintained. A stale claim-lock directory (`$WINGMAN_HOME/watch.pid.lock` or its owner-keyed equivalent) is blocking every arm attempt - remove it (e.g. `rmdir` that directory, after confirming no genuine `watch-fleet` arm is concurrently in progress) before resuming; re-arming alone will not succeed. This is a known bug (see the issue referenced in this plan's Open questions) - the lock is supposed to self-clear after 50 failed attempts and does not."*
- Any other hint (`sigkill-suspected` / `clean-exit-or-sigterm` / `hung-or-stale-pidfile`): the generic instruction from the finding-2 fix above applies unchanged - *"...resume it by running `/watch` again or arming `bin/watch-fleet` directly."*

`watch-spurious.log` remains useful as a persistent forensic history (multiple hints over time, for a human doing after-the-fact diagnosis) but is no longer anything `/watch` itself needs to read to pick the right remedy - the one hint that matters for the *current* trip is already sitting in the text `/watch` just received back from the command it ran.

**Filed separately, not fixed here:** the underlying bug - `watch-fleet:255-258`'s comment claims a stale lock "is cleared by the next arm once its 50 tries are exhausted," but `watch-fleet:290`'s actual `wm_die` never `rmdir`s anything, so the claim is false and the lock is permanently leaked until a human intervenes - is filed as **[issue #74](https://github.com/greerviau/wingman/issues/74)**, independent of this plan, per the fourth-pass review's original flag that it was "worth its own issue." This plan's `--classify` now correctly *detects* the failure and tells the pilot the *actual* remedy; it does not make the failure *self-healing*, since fixing that is a change to the claim-lock logic itself, not to the classifier built on top of it.

#### Finding 4 fix (r5): the exit-record's write side needs a priority order, not last-writer-wins

The third pass's finding 2 covered the *read* side of "one record slot, more than one writer" (a wake with no record to read, closed by the `cycle_live` check above). The fourth-pass review found the opposite direction still open on the *write* side: **two writers, one slot, wrong writer wins.**

Concretely: cycle A is live; wingman places a redundant arm B (the ordinary case `healthy` exists to describe). A fires - `fire()` writes `fire` to the record and has **already acked every surfaced event** (`watch-fleet:434-436`) before it reaches its own `rm -f "$PIDFILE"`. In the narrow window between A's record write and A's `rm`, B evaluates the singleton guard: A's pid is still alive and its beacon still fresh, so `cycle_live` is still true for B → under the r3/r4 design, B would write `healthy`, **overwriting A's `fire`**, and exit. Wingman then classifies A's wake, reads `healthy` (not `fire`), and does nothing - no report, no re-arm. The crew event is never surfaced, and because `fire()` already acked it, nothing will re-surface it later either: this is F1's original failure class (a genuine event silently swallowed) re-entering through the record slot itself.

**Fix, already folded into the F3 fix's write points above:** the `healthy` write is **non-clobbering** (skip it if a record already exists), while `fire` and `remote-control-dropped` are **unconditional**. This is sound because `healthy` is the only outcome the `cycle_live` check on the read side can already re-derive independently, from "record absent + a cycle is live" - losing a `healthy` write costs nothing, since finding 2's fix produces the same conclusion by a different, race-proof path. A `fire` or `remote-control-dropped` record, by contrast, is the *only* trace of an event that is either already acked (`fire`) or otherwise cannot be reconstructed after the fact, so it must never lose to a `healthy` write arriving later. The resulting priority is `fire` = `remote-control-dropped` (mutually exclusive within one process's own lifetime, so they never actually race each other) both strictly outrank `healthy`, never last-writer-wins.

#### Finding 5 fix (r5): the claim-time stale-record clear becomes loud, not silent

r3 introduced the claim-time clear as "cheap" defense-in-depth, reasoning that under the documented one-arm-one-classify protocol it should never fire. The fourth-pass review found the reasoning backwards: because `--classify` deletes every record it successfully reads, **the only way a record can still be present at a fresh claim is that a wake was never classified** - which is exactly the case where the record is the last remaining trace of an event `fire()` has already acked (the same permanent-loss shape as finding 4, reached from the opposite direction). A silent clear at that point does not defend against staleness; it destroys the one piece of evidence that a wake was missed. This is concretely reachable through the plan's own recovery instruction: `/watch`'s `spurious-repeated` message tells the pilot to resume supervision by arming directly, and a fresh arm run that way (not itself preceded by a fresh classify of whatever came before) would silently wipe out a pending record if one happened to exist.

**Fix (already folded into the F3 fix's claim-time write point above):** keep the clear, but make it loud - append the record's contents to `$WINGMAN_HOME/watch-spurious.log` as a `dropped-wake` line *before* clearing it, so a lost wake leaves forensic evidence rather than vanishing without a trace. This alone resolves the actual harm finding 5 identifies (silent, *untraceable* loss), without requiring `/watch`'s skill body to unconditionally classify before every arm - a stronger version of the fix the fourth-pass review also suggested, but one this plan deliberately does not adopt: making classification mandatory before *every* arm, including a fresh owner's very first one, would need `--classify` to distinguish "genuinely never armed before" from "armed before and something is pending," which is exactly the kind of additional wall-clock-adjacent state-inference this revision is trying to stop introducing after three rounds of exactly that going wrong. Logging the drop at the point it actually happens (`bin/watch-fleet`'s own claim-time code, which already knows unambiguously whether a record exists) is simpler and requires no new distinction anywhere.

**Residual, stated precisely rather than implied away (r6, fifth-pass nit):** "loud" means *traceable*, not *recovered*. If the dropped record was a `fire`, the crew event it represents is still permanently gone - `fire()` already acked it (`watch-fleet:434-436`) before writing the record, so nothing will re-surface it on its own, and `watch-spurious.log` has no reader unless a later `spurious-repeated` happens to send someone there. A `dropped-wake` line turns a silent, unexplained gap in supervision into an explainable one after the fact; it does not turn a swallowed crew event back into a delivered one. This residual is carried into Open questions below rather than left as an implicit claim in this section.

#### Finding 3 fix (r3): `remote-control-dropped` gets its own record, resolving the separator problem structurally

r2's classifier scanned raw stdout up to a literal `--` separator; the `remote-control-dropped` path never prints one (it writes its own two-line wake file and exits directly, `watch-fleet:451-460`), so an implementation keying on the separator's presence rather than "no separator found means scan everything" could silently misclassify wingman's own dropped Remote Control connection as `spurious` - exactly the class of silent swallowing the original F1 finding was raised to prevent, and the r2 testing strategy never exercised this shape.

The r3 exit-record redesign resolves this structurally rather than needing a patched scanning rule: `remote-control-dropped` now writes its own explicit record (see F3 fix, point 4 above), so there is no text to scan and no separator to find or miss. What remains is making sure the record actually gets written at that exit point (a one-line addition to `bin/watch-fleet`) and explicitly testing it - see the testing strategy below, which now lists this as its own case rather than assuming it is covered by the generic `fire` shape.

**Revised `/watch` design:**

```
---
description: Process one watch-fleet wake and arm the next cycle
allowed-tools: Bash(bin/watch-fleet:*), Bash(bin/crew-list:*), Read(~/.wingman/wake)
---
```

**Scope note (review F7):** this grant is correct only for wingman's own top-level session (owner `""`, the bare `~/.wingman/wake` path). A lead's watcher is keyed as `wake-<owner>`, which this grant does not cover. r1's `argument-hint: [--owner <lead-id>]` implied lead support that the grant never actually provided; this revision drops that argument-hint and scopes v1 to wingman's own top-level session only. `bin/watch-fleet --classify` itself is still built `--owner`-aware (it is trivial to make it so, and a lead's own future `/watch` would need it), but wiring a lead-scoped skill variant is out of scope for this spec and left as a named follow-up rather than half-implemented here.

Body, in order:

1. **If invoked in response to a `watch-fleet` background task's completion:** run `bin/watch-fleet --classify` (bare - no stdin, no pipe; see the F3 fix below for why) and act on the result:
   - `healthy` → a cycle is already live. Do nothing further - no report, no log, no re-arm. End the turn.
   - `fire` → a genuine crew event. Read `$WINGMAN_HOME/wake` (or `bin/crew-list` as the documented fallback) for the full roster, report a compact status to the pilot exactly as `CLAUDE.md`'s "Report" step specifies, then proceed to step 2.
   - `remote-control-dropped` → wingman's own Remote Control connection dropped. Relay `$WINGMAN_HOME/wake`'s message to the pilot immediately (run `/remote-control` to restore it), then proceed to step 2.
   - `spurious <count> <hint>` → `--classify` has already recorded the forensic detail (the same hint is on this line, but nothing branches on it here - it matters only for `spurious-repeated`, below). Report nothing to the pilot (nothing about the fleet changed), then proceed to step 2 immediately.
   - `spurious-repeated <count> <hint>` → the watcher has died `<count>` times in a row with no successful cycle in between (see the finding-2 fix below - this is a pure consecutive count with no wall-clock term at all, so it trips regardless of how slowly or quickly the failures recur, and regardless of whether any of them ever reached a successful claim; the trip also resets the count, so a lone stray death long after a genuine recovery does not immediately re-trip this). **Do not proceed to step 2.** Report this to the pilot as a genuine attention event, interpolating the real `<count>` field (r8 fix - never `WM_SPURIOUS_BUDGET_COUNT`, which the skill has no grant to read and which does not equal the trip count when overridden), with the remedy conditioned on `<hint>` - the third field `--classify` just printed on this same outcome line, not a separate read of `watch-spurious.log` - see the finding-3 fix below for the exact wording, since the generic "just re-arm" instruction is a known no-op for the `stale-claim-lock` hint specifically.
2. **Arm one fresh cycle, but only if none is already live and the failure budget was not just exceeded.** This is enforced twice for the ordinary case, deliberately: the `healthy` branch above already short-circuits before reaching this step, and `bin/watch-fleet`'s own singleton claim-then-check (`watch-fleet:277-305`) is atomic regardless, so arming here is always safe to *attempt* even under a race. `spurious-repeated` is a third, distinct reason to skip this step - not a race, but a deliberate refusal to keep re-arming a watcher that has just demonstrated it cannot stay up.
3. End the turn once armed (or once step 1 concluded no re-arm is warranted). Never call `/watch` twice in the same turn, and never bundle its arm onto the tail of another command.

#### F4 fix: the guard denies the `Skill` tool, so the gate-time path stays the raw Bash command

`hooks/pilot-preferences-guard.sh` branches on `tool_name` (`pilot-preferences-guard.sh:129`): `AskUserQuestion` is always allowed, `Read` is allowed only for the exact `$WM_HOME/wake` path, and `Bash` is allowed only when every segment resolves to one of a fixed set of shapes (`crew-list`, `watch-fleet`, `crew-ask await`, and the `prefs-list`/`pref-get`/`pref-set` forms). There is no branch for `Skill` at all (confirmed by reading the full script to its final fallthrough), so any `Skill` tool call - which is what invoking a project slash command actually is in this harness - reaches the guard's unconditional final `deny()` while preferences are pending.

This means `/watch`, if `CLAUDE.md`'s wake-loop section pointed at it unconditionally, would be **denied** during the pending-preferences window - exactly the window `docs/analysis/2026-07-13-unattended-boot-launch-behavior.md` documents relying on continued fleet supervision through the raw `bin/watch-fleet` Bash form.

**Fix:** `CLAUDE.md`'s wake-loop section, when updated to reference `/watch` as the normal-operation path, must explicitly retain the existing raw-command exemption for the gate-pending window: *while onboarding preferences are unanswered, arm and process `bin/watch-fleet` via the raw `Bash` form directly (already exempted by the guard); switch to `/watch` once preferences resolve (the common case, resolved once at the very start of a run before most work happens).* This requires no change to the guard at all, is the lower-risk of the review's two suggested options, and preserves a fallback path that is already tested and working rather than introducing new guard surface for `/watch`'s sake. (`/prefs` has no such fallback and needs the guard changed - see below.)

**Correction (r3):** r2 asserted this fallback covered both arming *and* processing, but at the time that was only half true - "processing" meant piping captured stdout into `--classify`, and no piped/heredoc'd command matches the guard's bare-basename exemption (see finding 1 under `/watch`'s F3 fix below). The r3 redesign (`--classify` as a bare `bin/watch-fleet --classify` command, no stdin) is what actually makes this fallback complete: arming and classifying are now both plain `watch-fleet` invocations, so the single raw-command instruction above genuinely covers the whole gate-time cycle, not just half of it.

#### F5 resolution: the explicit timeout recommendation

r1 recommended always arming with `timeout: 600000` and called it "free." The review required an empirical test before either keeping or dropping that recommendation, since the 300s test alone couldn't distinguish "the parameter is ignored for background tasks" from "the parameter is honored, and setting it to 600000 kills a quiet watcher every 10 minutes forever."

**Test performed:** armed `sleep 700 && echo SLEEP_DONE_700` via `Bash` with `run_in_background: true` **and an explicit `timeout: 600000`** (the exact value r1 recommended), specifically to observe whether the task dies at the 600s mark or survives to natural completion at 700s.

**Result: the task ran the full 700 seconds and completed cleanly (exit 0, `SLEEP_DONE_700` captured in full)** - 100 seconds past the explicit 600000ms timeout that was set. Confirmed directly against the completed background task (`bc4pn8mbu`, armed 2026-07-13T20:01 UTC, completed ~20:13 UTC).

**Conclusion and final recommendation: the `timeout` parameter does not govern a `run_in_background: true` task's actual lifetime.** Setting it is not a "defensive floor" - it is inert. r1's claim that specifying `timeout: 600000` was "free" and defensive was half right (it is not actively harmful, unlike the review's worst-case worry of a 10-minute self-kill cadence) and half wrong (it buys nothing; there is no floor being defended). **This revision drops the explicit-timeout recommendation entirely** rather than keep a rationale now known to be false: `/watch` should arm `bin/watch-fleet` with no `timeout` parameter, exactly as `CLAUDE.md` already documents ("e.g. Bash `run_in_background`," no timeout mentioned), and the real question of whether *any* upper bound exists on a backgrounded task's lifetime (tested clean to 700s now, twice with two different parameter combinations) stays open only in the sense that no failure has yet been found, not in the sense that one is expected. Combined with the rest of this investigation, this further strengthens the conclusion that the confirmed, precedented mechanism (a self-inflicted kill, issue #64) - not a harness-level timeout of any kind - is what actually explains the observed incident.

### 3. `/prefs` - blocked, deferred until the guard is extended

**Worth building?** The review confirmed r1's own honesty that this is marginal (the hard failure mode - skipping preferences entirely - is already mechanically closed by `hooks/pilot-preferences-guard.sh`), and found a further, genuinely blocking problem r1 missed entirely: **as specified, `/prefs` cannot run at all.**

#### F4 fix (the `/prefs`-specific half): the guard must be extended, narrowly, for this one skill

Per the same guard mechanics described under `/watch`'s F4 fix above: a `Skill` tool call is not on `pilot-preferences-guard.sh`'s allowlist, so invoking `/prefs` is denied by the guard's unconditional fallthrough `deny()` - during the *only* window `/prefs` has any purpose. Unlike `/watch`, there is no raw-command fallback to fall back on, because `/prefs`'s entire job (calling `AskUserQuestion` with the batched, still-missing questions, then `pref-set` each answer) only exists as skill logic - it was never a bare `bin/` command wingman ran directly.

**Fix:** extend `hooks/pilot-preferences-guard.sh` with one additional, narrow branch: `tool_name == "Skill"` **and** the invoked skill name is exactly `prefs` → `allow()`. This is a deliberate, minimal expansion of the guard's trust surface - scoped to this one named skill, not a blanket allowance for any `Skill` call - consistent with how the guard already narrowly allows exactly `crew-list`/`watch-fleet`/`crew-ask await` as bare Bash shapes rather than trusting Bash broadly. **Before implementing this, confirm the exact `tool_input` field the harness emits for a `PreToolUse` event on the `Skill` tool** (the field naming the invoked skill, e.g. `skill`) - this spec does not assert that name with certainty, since no existing hook in this codebase currently inspects a `Skill` tool call to crib from. This is the one prerequisite `/prefs` is blocked on; it should ship together with, not before, the guard change.

#### F10 fix: the `allowed-tools` shape

r1's frontmatter named `Bash(uv run --no-project --quiet bin/lib/wm-state.py prefs-list:*)` (relative path) but reasoned about verifying it against `$WINGMAN_STATE`, which expands to the **absolute** path `/home/agents/github/wingman/bin/lib/wm-state.py` on this install - a mismatch that would never actually match at grant-check time. The fix is not to hardcode the absolute (and install-specific) path; it is to have `/prefs`'s body invoke the command in the same **relative**, cwd-based form every other skill already uses (`uv run --no-project --quiet bin/lib/wm-state.py ...`, matching how `/spawn` calls `bin/spawn-crew` rather than an absolute path), so the grant and the invoked command agree by construction:

```
---
description: Confirm onboarding preferences (remote, artifact_linking, verbosity) once per run
allowed-tools: AskUserQuestion, Bash(uv run --no-project --quiet bin/lib/wm-state.py prefs-list:*), Bash(uv run --no-project --quiet bin/lib/wm-state.py pref-set:*)
---
```

`AskUserQuestion` is now listed explicitly (r1 omitted it from `allowed-tools` while the body called it - harmless only if `allowed-tools` is an additive grant rather than a restrictive whitelist; the implementing developer should confirm which behavior applies before shipping and this spec now lists it either way rather than leaving that ambiguous). r1's second F10 point - that `cmd_match.py` and Claude Code's own permission engine are different matchers with different rules, and that "verify against `cmd_match.py`" was a category error - is accepted as correctly identified by the review; verification of this skill's `allowed-tools` shape belongs against Claude Code's permission engine specifically, and separately, if the guard change above is implemented, its `Skill`-name matching belongs to the guard's own Python logic, not `cmd_match.py` either (the guard's existing `Bash`-segment matching is the only place `cmd_match.py` is actually relevant).

Body (unchanged from r1's logic, now runnable once the guard change lands):

1. If `$WINGMAN_RUN_ID` is unset, do nothing and say nothing (matches the documented conservative-default branch for a session not launched via `bin/wingman`).
2. Run `prefs-list --run-id "$WINGMAN_RUN_ID"`; diff against the three required keys (`remote`, `artifact_linking`, `verbosity`).
3. If nothing is missing, do nothing.
4. Otherwise, say "Before I start working, I need to ask you some preference questions:" and call `AskUserQuestion` **once**, with only the still-missing questions batched into that single call - never split across multiple calls - then `pref-set` each answer.

## Testing strategy

- **`bin/watch-fleet` exit-record writes**: confirm each of the three write points actually writes the expected token before its existing exit - `healthy` (singleton early-exit, non-clobbering per r5 finding 4), `fire` (inside `fire()`, unconditional), and, explicitly and separately (previously untested until r3), `remote-control-dropped` (the `self_pane_check` exit path, which never calls `fire()` and has no `--` separator to accidentally rely on, unconditional). Confirm the fresh-claim-time behavior is the r5 "loud" version: a pending record is appended to `watch-spurious.log` as `dropped-wake` *before* being cleared, not silently discarded.
- **`--classify`'s dispatch point** (new, r4 finding 3): assert that invoking `--classify` while a cycle is live neither creates the claim-lock directory nor mutates `$PIDFILE`/`$BEATFILE` - the direct regression test for a developer accidentally letting `--classify` fall through to the arm path instead of joining the `--status`/`--stop` mode block.
- **`bin/watch-fleet --classify` outcome classification** (r4 redesign): unit-test against a real exit-record file for `healthy`, `fire`, and `remote-control-dropped` (record present), and confirm the record is deleted after being read (a second consecutive `--classify` call with nothing new having run must not replay the same outcome). **New for r4 finding 2:** with the record absent, test both branches of the added `cycle_live` check - a live cycle (record absent because it hasn't exited yet, e.g. the redundant-arm race the finding traces) must classify `healthy`, with no log write and no budget tick; only a dead cycle reaches the hint logic. Separately test the now-exhaustive pidfile forensic hint on the genuinely-spurious path: a pidfile naming a dead pid → `sigkill-suspected`; an absent pidfile → `clean-exit-or-sigterm`.
- **The failure budget** (r5 redesign, corrected in r6: a pure consecutive count, no lifetime qualifier, no wall-clock term anywhere, exactly three mutation rules all inside `--classify`): the direct regression test for r4 findings 1 and 2 must prove the budget's trip condition depends on nothing but the *sequence* of outcomes, never on elapsed time. Concretely: (a) three genuinely spurious classifications in a row, **each separated by an arbitrary, even large, delay both between the cycle's death and its classification and between classify calls themselves** (simulating both a slow death and a slow wingman turn - this is what r3's and r4's own tests each failed to do, and is the specific thing that let each of their defective designs pass), must still return `spurious-repeated` on the third; (b) **the plan's own named repro - a stale `$PIDFILE.lock` directory in place, so three consecutive arms die at `watch-fleet:290` without ever claiming, never writing an exit-record - must reach `spurious-repeated` on the third** (the direct regression test for r4 finding 2: a death that never claims must count exactly like any other spurious death, not fall through an "unknown lifetime, don't count" gap); (c) a `healthy` or `fire` classification between two spurious ones must reset the count, so a fourth spurious after that reset needs a fresh 3, not 1; (d) **new for r6 finding 2:** after `spurious-repeated` fires, a single subsequent spurious classification with no intervening non-spurious outcome must report plain `spurious`, not `spurious-repeated` - the direct regression test for reset-on-trip, and the one that would have caught the fifth-pass review's "sticky forever after one trip" defect; (e) **new for r6 finding 1, the most direct regression test in this whole plan:** an arm that successfully claims (`echo $$ > "$PIDFILE"` succeeds) and is then killed must **not** alter `watch-spurious-count` at the moment of claiming - increment the count to some nonzero value via prior spurious classifications, then let a fresh arm claim successfully and die immediately after, and confirm the count is unchanged by the claim itself (only `--classify`'s own subsequent read of that death may change it). This is the test that would have caught the "Files touched" instruction contradicting the design section, and it must be run against the actual `bin/watch-fleet` claim-time code path, not just against `--classify` in isolation.
- **The now-exhaustive hint logic** (r5 finding 3, extended in r6, ordering corrected in r7): in addition to the two pidfile-residue cases, test the third state directly - a pidfile present, its pid alive, but its beacon stale beyond `$GRACE` - and confirm it returns `hung-or-stale-pidfile` rather than aborting `--classify`. With a `$PIDFILE.lock` directory present (simulating the leaked-claim-lock repro) and no cycle live, confirm `--classify` returns the `stale-claim-lock` hint in preference to whatever the pidfile residue would otherwise suggest. **New for r7 finding 1, the direct regression test for the Files-touched ordering contradiction:** with a `$PIDFILE.lock` directory present **and a cycle genuinely live** (a stale, unrelated leaked lock coexisting with a fresh, healthy, successfully-claimed cycle), confirm `--classify` returns `healthy` - **not** `spurious`/`stale-claim-lock` - proving `cycle_live` is checked, and wins, before the lock check is ever reached.
- **The hint (and, since r8, the count) reach `/watch` without any file read** (new, r7 finding 2; extended, r8 should-fix 1): confirm `--classify`'s own stdout for both spurious outcomes is a single line carrying the count and the hint (`spurious 1 sigkill-suspected`, `spurious-repeated 3 stale-claim-lock`, etc.), and confirm `/watch`'s remedy selection and its pilot-facing count are driven entirely by parsing that captured line - not by a `Read` of `watch-spurious.log`, which `/watch`'s `allowed-tools` does not and should not need to grant. Confirm `/watch`'s `spurious-repeated` report interpolates the real count field (never `WM_SPURIOUS_BUDGET_COUNT`) and carries the lock-specific remedy (naming the lock directory to clear) when and only when the hint on that line is `stale-claim-lock`, and the generic "just re-arm" remedy for every other hint.
- **Record write priority** (r5 finding 4): reproduce the exact interleaving the fourth-pass review traced - a live cycle A about to fire, a redundant arm B evaluating the singleton guard while A's `fire` record is already written but before A's `rm -f "$PIDFILE"` - and confirm the record still reads `fire` afterward, never clobbered by B's `healthy`. Confirm the general rule: a `healthy` write is skipped whenever a record is already present; a `fire`/`remote-control-dropped` write is never skipped.
- **Loud claim-time drop** (r5 finding 5): arm a cycle, get it into a state where a record is pending and unconsumed (no classify call in between), then arm again; confirm the record's contents appear as a `dropped-wake` line in `watch-spurious.log` before the record is cleared, rather than vanishing silently.
- **`--classify`'s own argument acceptance** (r5 nit): confirm `bin/watch-fleet --classify` and `bin/watch-fleet --classify --owner <id>` are accepted by the top-level argument parser (`watch-fleet:82-90`) and do not hit `wm_die "unknown arg"` - a regression test distinct from the dispatch-point test, since the dispatch block and the argument parser are two different places in the file that both had to learn about `--classify`.
- **`/watch`**: exercise against a live cycle end-to-end - (a) let a cycle fire naturally on a real crew status transition and confirm the skill reports a roster and re-arms; (b) arm while a cycle is already live and assert the skill ends the turn with no report, no log write, and no re-arm (the direct regression test for F2's `healthy` path, and, via the `cycle_live` check, also for a record-absent-but-live-cycle race); (c) `SIGKILL` and `SIGTERM` a live cycle separately (they leave different pidfile residue) and confirm both are classified `spurious` with the correct hint and the skill re-arms silently; (d) drive the timing-independent budget test above through the actual skill and confirm it reports `spurious-repeated` with the recovery instruction and does **not** attempt a further arm. Confirm the command actually run at classify time is a bare `bin/watch-fleet --classify` (no pipe, no heredoc) against both the skill's own `allowed-tools` grant and, separately, `hooks/pilot-preferences-guard.sh`'s matcher while preferences are pending - the direct regression test for r3 finding 1.
- **`/say`/`/ask`**: confirm each against a live crew member with no prior permission grant present, verifying no permission prompt fires. For `/ask`, confirm the two-phase flow end-to-end including the `--answer-file` path added for F6 (a delegate reply that uses `--answer-file` should trigger the skill's follow-up read of `~/.wingman/ask/<req>.json`, not just relay the truncated inline text).
- **`/prefs`**: confirm the three branches (nothing missing → no-op; `$WINGMAN_RUN_ID` unset → silent no-op; something missing → exactly one batched `AskUserQuestion` call, not three), and confirm the new guard branch allows exactly `Skill` calls to `prefs` while preferences are pending and nothing broader (a `Skill` call to some other skill name must still be denied during the gate).

## Open questions / risks

- **F5 is resolved**: an explicit `timeout: 600000` was empirically confirmed inert against a `run_in_background: true` task (700s test, completed clean past the 600s mark) - the explicit-timeout recommendation is dropped, not kept as a "harmless defensive floor." (Per the r3 re-review's own minor note: this is one observation, not a settled property of the harness - the *instruction* it produces, arm with no explicit timeout, is safe either way, so it does not need re-testing before implementation.)
- **The exact `PreToolUse` payload field naming the invoked skill is unconfirmed** (`/prefs`'s guard fix) - verify against the harness's actual hook input shape before implementing, not by assertion.
- **The r3 exit-record redesign replaces a text-pattern drift risk with a smaller one.** r2's risk was two independently-maintained prefix lists (`fire()`'s prefixes and `--classify`'s regex). r3's is a single-word vocabulary (`healthy`/`fire`/`remote-control-dropped`) written by `watch-fleet` itself at exactly three exit points and read verbatim by `--classify` - there is no pattern-matching left to drift, only the discipline of adding a fourth write point if `watch-fleet` ever grows a fourth kind of clean exit. This is a smaller, more mechanical risk than r2's, worth naming rather than assuming away.
- **The failure-budget threshold (`WM_SPURIOUS_BUDGET_COUNT`, default 3) is a starting point, not a tuned value**, but is env-overridable by design, matching `WM_STALL_IDLE`/`WM_PERM_LIVENESS_GRACE`/etc. elsewhere in `bin/watch-fleet` - so it can be tuned post-deployment without a code change. There is deliberately no second threshold (r4's `WM_SPURIOUS_MIN_LIFETIME` is removed along with the lifetime qualifier it gated - see the finding-2 fix) - the r5 design has exactly one tunable knob for this budget, not two.
- **The pure consecutive-count budget accepts a known, named false-positive class in exchange for closing a false-negative class that survived three prior revisions.** This tradeoff is stated explicitly in the finding-2 fix above, attributed to the fourth-pass review's own offered alternative and accepted assessment, not discovered independently; the r6 reset-on-trip rule narrows it further (only three *quick, unrepaired* consecutive failures trip it, never a lone stray death long after a genuine recovery) - worth flagging here too since it is the one deliberate, load-bearing judgment call in this revision rather than a bug fix.
- **A dropped `fire` record is a permanently lost crew event, not merely a logged one.** The r6 finding-5 fix makes the loss *traceable* (a `dropped-wake` line in `watch-spurious.log`), not *recoverable* - `fire()` has already acked the event before the record it wrote can ever be dropped, so nothing re-surfaces it automatically, and the log itself has no reader unless a later `spurious-repeated` happens to send someone there. This is a narrow window (the claim-time drop only triggers when a record was left genuinely unconsumed) but the failure mode, if hit, is silent from the pilot's perspective until they happen to read the log - worth monitoring in practice rather than assuming the forensic trail alone closes the gap.
- **[Issue #74](https://github.com/greerviau/wingman/issues/74)** (the leaked claim-lock directory is never actually cleared, contrary to `watch-fleet:255-258`'s own comment) is a **new** open prerequisite alongside #64 and #67, filed during this revision: `--classify`'s `stale-claim-lock` hint now correctly *detects* this failure and gives the pilot a working manual remedy, but the underlying bug - a lock that should self-clear after 50 failed attempts and does not - is unfixed, and every occurrence still requires manual intervention until it lands.
- **No cleanup lifecycle exists for any owner-keyed `watch-fleet` state on stand-down** (`watch.pid`/`.beat`, `wake`, and now `watch-exit`/`watch-spurious-count`, all `-<owner-key>`-suffixed for a lead) - confirmed by inspection: `bin/crew-standdown` does not reference any of these paths today. This is a pre-existing gap this plan does not introduce and is out of scope to fix here, but the two new files this plan adds should be included in whatever future work closes it, rather than assumed to already be handled.
- **Issue #64 (self-kill guard) and issue #67 (stuck-loop root cause) are both open prerequisites** this spec depends on being tracked to completion, not `/watch` substitutes for either. Note the r3 finding-2 defect (`spurious` re-arming unconditionally) and its three subsequent, increasingly narrow near-misses (r4's still-timing-dependent budget; the "dies before claiming" gap; r5's claim-time-reset contradiction and quiet-fleet stickiness) are all plausible mechanisms for the very stuck-loop incident issue #67 exists to root-cause, alongside F2's original `healthy`-path mechanism - all are now closed in this spec, but issue #67's own reproduction is still the thing that would confirm which, if any, was the actual cause, not this spec's fixes by themselves.
- **The harness-side reap-across-compaction hypothesis (review F9) remains untested.** If `/watch`'s `spurious` classification, once shipped, shows a pattern in `watch-spurious.log` correlated with session compaction events rather than with `sigkill-suspected`/systemd-timer timing, that would be the confirming signal for this hypothesis - worth revisiting once real data exists rather than testing it synthetically now.
