# Spec: supporting agent CLIs beyond Claude Code (opencode, pi, codex, grok)

- **Date:** 2026-07-11
- **Type:** General spec — requirements and approach (analyst deliverable; a follow-up architect turns this into a detailed implementation plan)
- **Repo:** `wingman`
- **Components in scope:** `bin/spawn-crew`, `bin/lib/common.sh`, `bin/crew-takeover`, `bin/watch-fleet`, `bin/doctor`, `CLAUDE.md` ("Harness-agnostic by design"), `tests/`
- **Out of scope:** the detailed per-CLI flag wiring, the adapter file format's exact schema, and code changes — those belong to the architect/developer phase that follows approval of this spec.

## 1. Problem

Wingman launches every crew member the same way: an independent `claude` session in a tmux window, driven by `bin/spawn-crew`.
CLAUDE.md documents `WM_AGENT` as "the single place to change for a different harness," but that claim is narrower than it reads.
Concretely, in `bin/spawn-crew` today:

- `WM_AGENT` (default `claude`) only substitutes the **binary name**; every flag emitted after it is Claude-Code-shaped and hardcoded: `--permission-mode <mode>`, `--session-id <uuid>`, `--model`, `--effort`, `--add-dir` (repeated), `--append-system-prompt "$(cat <file>)"` (`bin/spawn-crew:150-172`).
  Pointing `WM_AGENT` at a different binary without also rewriting these flags produces a broken launch line, not a working alternate harness.
- The swap is **fleet-wide and global**: one environment variable picks one CLI for every crew member spawned for the life of the process.
  There is no way for one crew member to run on `codex` while another runs on `claude` in the same fleet — every `--model`/`--effort`-style per-spawn knob wingman already has is per-invocation, but `WM_AGENT` is not.
- Two other places already carry Claude-Code-specific assumptions that `WM_AGENT` does not touch at all:
  - `bin/crew-takeover`'s dead-window recovery hint is a literal string: `"cd $REPO && <your-agent-cli> --resume $SID"` with a worked example naming `claude --resume $SID` (`bin/crew-takeover:31-34`). It gestures at being generic (`<your-agent-cli>`) but the concrete example, and the assumption that `$SID` is resumable at all, are Claude Code's.
  - `bin/watch-fleet`'s prompt-freeze detector — the regex family `WM_PERM_PROMPT_RE`/`WM_PERM_OPTION_RE`/`WM_PERM_LEAD_RE` that recognizes a crew member frozen on an interactive permission or trust dialog — is explicitly "verified against a live capture, Claude Code v2.1.206" (`bin/watch-fleet:162-190`). It is overridable via env vars, but there is exactly one override, so it still cannot describe four different CLIs' dialog shapes at once.
- `bin/doctor` hardcodes `claude` as an irreducible, always-required dependency (`bin/doctor:48`), independent of which CLI a fleet actually uses.

None of this is a defect in what shipped — `WM_AGENT` and the tmux-only takeover/crew-say path were deliberately scoped to keep the crew-coordination layer (tmux windows, `crew.json`, the watcher, the board) harness-neutral, and they succeed at that.
The gap is one layer down: **there is no per-CLI adapter**, so every harness swap today means hand-editing `bin/spawn-crew`'s flag-composition code, and two of the three places that encode CLI-specific behavior (`crew-takeover`'s resume hint, `watch-fleet`'s freeze detector) are not reached by `WM_AGENT` at all.

The requester wants wingman to run crew members on **opencode, pi, codex, and grok** in addition to Claude Code, with the mechanism open to more CLIs later without repeating this exercise per addition.

## 2. What wingman actually needs from an agent CLI

Reading `bin/spawn-crew` and `bin/lib/common.sh` end to end, a crew member's agent CLI is asked to do exactly seven things.
Everything below is graded against these seven, per target CLI:

1. **Launch as a long-lived interactive session** in a tmux pane and keep running across many turns (not a one-shot prompt-and-exit). Wingman never uses a CLI's non-interactive/one-shot mode for the crew member itself — the objective is delivered *into* the interactive session by simulated typing (`wm_tmux_send_message`), not as a launch argument. This part of the design is already fully harness-neutral: any interactive TUI that accepts typed input and Enter as a real user turn satisfies it, which is true of all four target CLIs.
2. **Run unattended without hanging on a permission prompt** — Claude Code's `--permission-mode bypassPermissions`. No human sits at a crew member's terminal, so a CLI that pauses on a per-tool-call approval with no equivalent bypass flag will freeze forever unless `watch-fleet`'s freeze detector catches it and the pilot manually approves via `crew-takeover`.
3. **Accept (or approximate) a caller-chosen session identifier at launch** — Claude Code's `--session-id <uuid>`, generated by `spawn-crew` and stored in `crew.json` as `session_id`, later used by `crew-takeover`'s resume hint for a dead window.
4. **Receive the crew brief** (playbook + status contract, composed into `$SYSPROMPT`) as context before the first real turn — Claude Code's `--append-system-prompt "$(cat file)"`.
5. **Select a model** (`--model`) and, where the CLI distinguishes it, a reasoning/thinking effort (`--effort`).
6. **Optionally carry a human-readable display name** distinct from the tmux window name — Claude Code's `-n`/`--name`, added in the (not-yet-merged) branch `feat/spawn-crew-name-flag` (PR #18). This is cosmetic, not load-bearing: nothing in the crew-coordination layer reads it back.
7. **Be observable via tmux pane text alone**, with no other status channel — the watcher's stall/freeze detection and `crew-say`/`crew-takeover` all work purely by reading and typing into the pane (`wm_tmux_pane_text`, `wm_tmux_window_activity_age`, `wm_tmux_pane_ready`). This is inherently true of any terminal program and needs no per-CLI work *except* item 2's freeze-detection signature, which is pattern-matched against the specific words and layout each CLI's dialog renders.

Items 1 and 7 are free — they hold for any interactive terminal CLI and require no adapter.
Items 2–6 are exactly the surface that needs a per-CLI mapping.

## 3. Per-CLI capability findings

Researched against each project's own current CLI docs (July 2026).
Treat exact flag spellings as a snapshot to re-verify at implementation time, not a frozen contract — these CLIs are actively developed and flags shift between releases.

| Need | Claude Code (baseline) | opencode | codex | grok | pi |
|---|---|---|---|---|---|
| Interactive launch | `claude` | `opencode` (TUI is the default) | `codex [PROMPT]` | `grok` (TUI is the default) | `pi ["prompt"]` |
| Bypass permission prompts | `--permission-mode bypassPermissions` | `--dangerously-skip-permissions` (docs flag it as disposable-VM-only), or partial `--auto` (auto-approves only requests not explicitly denied) | `--dangerously-bypass-approvals-and-sandbox` (alias `--yolo`), or `--ask-for-approval never` + `--sandbox workspace-write` | `--always-approve` (alias `--yolo`) | `-a`/`--approve` — trusts project-local files for the run; **docs describe this as trust-prompt handling, not confirmed as a full tool-call-approval bypass** |
| Force/choose a session id at launch | `--session-id <uuid>` (forced, wingman-chosen) | `--session <ID>` is documented as *resuming* an existing session; **no confirmed flag to force a caller-chosen ID for a brand-new session** | no documented flag to force a session id at creation; `codex resume [SESSION_ID]` only resumes an existing (codex-assigned) one | `--session-id <UUID>` — explicitly "assign a specific UUID to a new session" (direct equivalent) | `--session <path\|id>` resumes; **no confirmed flag to force a new session's id at creation** |
| Inject a system/brief prompt before the first turn | `--append-system-prompt "$(cat file)"` | not found in the CLI reference researched | not found in the CLI reference researched | not found in the CLI reference researched | not found in the CLI reference researched |
| Select a model | `--model` | `--model`/`-m` | `--model`/`-m` | `--model`/`-m` | `--model` (+ `--provider`, since pi is multi-provider) |
| Select reasoning/thinking effort | `--effort` | not found | via `/model` slash command mid-session; no launch flag found | not found | `--thinking off\|minimal\|low\|medium\|high\|xhigh\|max` |
| Display name | none merged yet (PR #18, branch only) | `--title` (session title) | none found; `/rename` is an interactive-only slash command | none found | `--name`/`-n` — "set session display name at startup" (direct equivalent) |
| Resume a dead session later | `claude --resume <sid>` | `--session <ID>` / `--continue` / `--fork` | `codex resume [SESSION_ID]` / `--last` / `--all` | `--resume [<ID>]` / `--continue` / `--fork-session` | `-r`/`--resume` (interactive picker) / `-c`/`--continue` / `--session <id>` |

Two findings matter more than any single flag:

- **The system-prompt-injection gap is universal.** None of the four target CLIs was found to have a documented equivalent of `--append-system-prompt`. Wingman's crew brief (playbook + status contract) is not optional context — it *is* how a crew member knows the status-file protocol, the escalation rules, and its own type's playbook. Every adapter needs an answer to "how does the brief reach the model before the objective," and the answer is likely CLI-specific (e.g., prepend the brief to the first typed message instead of passing it as a launch flag; or drop it into a project-level instructions file the CLI auto-loads, if one exists, analogous to this repo's own `CLAUDE.md` convention). This is a design question for the architect phase, not something this spec resolves, but it must be named as a hard requirement rather than discovered late.
- **"Grok" is ambiguous and must be disambiguated before implementation.** Research surfaced two distinct CLIs answering to that name: xAI's own "Grok Build" (`docs.x.ai/build/cli`, the source for `--session-id`/`--always-approve`/`-m` above) and a third-party open-source `grok-cli` (`superagent-ai/grok-cli`) wrapping the Grok API. They are not the same program and will not share a flag surface. The spec assumes xAI's own CLI is the intended target (it is the more actively maintained, first-party option with the closest flag parity to Claude Code's needs), but this should be confirmed with the requester before the architect designs against it.

## 4. Requirements

1. **Per-crew-member CLI selection**, not just fleet-wide. `spawn-crew` gains an `--agent <name>` flag (mirroring the existing `--model`/`--effort` per-spawn pattern), falling back to `$WM_AGENT`, falling back to `claude`. This lets one fleet mix CLIs — e.g. an analyst on `claude`, a developer on `codex` — without an env var juggling act between spawns.
2. **A data-driven adapter per CLI**, not branching logic inside `spawn-crew`. Each supported CLI gets one small descriptor (config, not code-with-control-flow) that maps wingman's seven abstract needs (§2) to that CLI's real flags: binary name, bypass-permission flag(s), session-id flag (or "unsupported"), model/effort flags, display-name flag (or "unsupported"), and how the brief is injected. This mirrors the existing `playbook/<type>.md` open-endedness: adding CLI #5 means adding one new descriptor file, not editing `spawn-crew`'s control flow. `spawn-crew --list-types` already exists for playbooks; an equivalent `--list-agents` (or similar) should exist for discoverability.
3. **Graceful degradation is a first-class behavior, not an afterthought**, for every capability a target CLI lacks:
   - *No bypass-permission equivalent found for a given CLI* → the adapter says so explicitly, and wingman's docs/warnings reflect that this CLI's crew members are more likely to freeze on a prompt (relying on the watcher's freeze detector + `crew-takeover`, not eliminating the freeze risk).
   - *No forced-session-id equivalent* → `crew.json`'s `session_id` field is populated with whatever identity is available (possibly none), and `crew-takeover`'s dead-window recovery message degrades to that CLI's own resume mechanism (e.g. "resume the most recent session in this directory" instead of "resume this exact id") rather than asserting a capability that doesn't exist.
   - *No display-name equivalent* → simply omit the flag; already established as non-load-bearing today.
4. **The freeze/prompt detector must become adapter-scoped, not left as a single Claude-Code-tuned regex.** `watch-fleet`'s `WM_PERM_PROMPT_RE`/`WM_PERM_OPTION_RE`/`WM_PERM_LEAD_RE` triple is calibrated against one CLI's UI text and shape. Today it is overridable by one global env var, which cannot hold four (or more) CLIs' dialog signatures simultaneously. Each adapter must carry its own freeze signature (or a documented "not yet characterized" placeholder), and `watch-fleet` must look up the signature for the specific crew member's agent rather than applying one global pattern to every window. This is the requirement most likely to be silently skipped and most likely to cause real harm if it is (a crew member frozen on a prompt in an unrecognized shape never gets flagged `blocked`, and the existing silent-stall backstop is not a substitute — a static dialog and a genuinely stalled process are different failure modes the current design deliberately tells apart).
5. **`crew-takeover`'s resume-recovery text must be adapter-driven**, not a hardcoded Claude Code example, consistent with requirement 3's degradation rule.
6. **`bin/doctor` must check whichever CLI(s) are actually in play**, not hardcode `claude` as irreducible. At minimum, check `$WM_AGENT`'s configured binary the same way it already conditionally checks `gh` only when the active developer playbook references it (`bin/doctor:71-76` is the existing precedent for "check what's actually configured, not everything possible").
7. **The system-prompt-injection answer must be decided per adapter** (§3), and must keep working through the existing large-text-paste handling in `wm_tmux_send_message` (already generic — it already tolerates a bracketed-paste burst plus a delayed Enter for arbitrarily large typed text, so prepending the brief to the first message, if that is the chosen fallback, does not require changes there).
8. **No change to the crew-coordination layer.** `crew.json`'s schema, the wake protocol, `board.md`, the status-file contract (`playbook/_status-contract.md`), and the team guardrail are all agent-CLI-agnostic today (they operate on tmux windows and status files, never on a CLI's own flags or output format) and must stay that way. The adapter mechanism is additive to `bin/spawn-crew`, `bin/crew-takeover`, `bin/doctor`, and `bin/watch-fleet` only — the same boundary CLAUDE.md already draws, made real rather than aspirational.
9. **Verify flags against the real binary before shipping an adapter**, not just documentation. All four CLIs are under active development; the capability table in §3 is a snapshot that should be re-checked (`--help` output, a smoke-tested launch) at implementation time rather than trusted as final.

## 5. Non-requirements (explicitly out of scope for this effort)

- Running a genuinely mixed fleet in the same tmux session is a side effect of requirement 1, not a goal to design further toward (e.g. no requirement for a unified cross-CLI status format beyond what the existing status-file contract already provides — that contract is authored by the crew member itself, not parsed from the CLI's output).
- Routing individual tool-call permission decisions up to the pilot as real per-decision blockers (rather than blanket bypass) is called out as an existing, separate follow-up in `bin/spawn-crew:86-88` already and is not entangled with this effort.
- A general plugin/extension API for third-party adapters beyond the four named CLIs. The mechanism should be *open to* more CLIs (one new descriptor file), but building a public contribution workflow for it is not requested here.

## 6. Proposed approach (shape, not detailed design)

Introduce one small **adapter descriptor per CLI** (exact file format — shell-sourceable key/value, or a small structured file read by the same `wm_py` helper `spawn-crew` already uses for JSON — is an architect decision) living alongside the playbooks, e.g. `bin/agents/<name>.sh` or similar, one per supported CLI (`claude`, `opencode`, `codex`, `grok`, `pi`, ...). `spawn-crew` resolves `--agent`/`$WM_AGENT`/`claude` to a descriptor the same way it already resolves `--type` to a playbook file (`bin/spawn-crew:76-82` is the existing pattern to mirror), then composes the launch line from the descriptor's fields instead of the current hardcoded flag list. `watch-fleet` and `crew-takeover` read the same descriptor (via the crew member's recorded agent name in `crew.json`, a new field alongside `session_id`) for the freeze signature and resume hint respectively, so all three touch points stay in sync off one source of truth per CLI instead of three independent hardcodings.

This keeps the change confined to exactly the files CLAUDE.md already names as the harness-specific seam (`bin/spawn-crew`, plus the two follow-on touch points this investigation found were *not* actually covered by `WM_AGENT` today: `bin/crew-takeover` and `bin/watch-fleet`), and keeps `bin/doctor` in sync via the same descriptor set.

## 7. Testing strategy

- The existing E2E suite substitutes a `sleep 60` stub for `$WM_AGENT` (`tests/spawn-scope.test.sh:17`), which validates that `spawn-crew` launches *something* and records roster state correctly, but does not exercise real per-CLI flag composition today. Extend this: a test per adapter descriptor that asserts the composed launch line contains the expected flags for that CLI (bypass-permission, model, session-id-or-absence) without needing the real binary installed — a stub script that just echoes/logs the args it received is sufficient, following the same pattern already used.
- A smoke test against each real CLI binary (where installed in CI or a dev sandbox) that launches, receives a trivial objective via `wm_tmux_send_message`, and confirms the pane produces output — this is the only way to validate the "verify against the real binary" requirement (§4.9) rather than trusting documentation.
- A dedicated freeze-detector test per adapter: capture (or hand-construct) a sample of that CLI's actual permission/trust dialog text and assert the adapter's signature matches it and does not false-positive on ordinary transcript content, mirroring the care already visible in `watch-fleet`'s comments about avoiding false positives on prose that merely mentions a prompt (`bin/watch-fleet:210-220`).

## 8. Risks / open questions for the architect phase

1. **System-prompt injection has no found equivalent in any of the four CLIs.** This is the single highest-impact open question — without it, a crew member on a non-Claude CLI does not receive its playbook or the status contract through the same mechanism, and an incorrect fallback (e.g. one that gets silently truncated, or arrives after the CLI's own project-file auto-load has already primed a different persona) could produce a crew member that never reports status correctly. Needs hands-on verification against each real binary, not just documentation, before the architect commits to one fallback strategy.
2. **Forced session IDs are only confirmed for grok (`--session-id <uuid>` explicitly documented as assigning a new UUID).** For opencode, codex, and pi, the CLI likely mints its own ID at creation and only accepts resuming *existing* IDs afterward. If so, `crew-takeover`'s "resume this exact session" story degrades to "resume the most recent session in this repo," which is weaker (ambiguous if a repo hosts more than one crew member on the same CLI) and needs a documented fallback identity strategy (e.g. capture the CLI-assigned id by having it write to a discoverable location, or reading its own session index, if one exists) — worth a design spike per CLI rather than an assumption.
3. **The permission-bypass equivalents vary in strength.** `--always-approve`/`--yolo` (grok), `--dangerously-bypass-approvals-and-sandbox` (codex) are unambiguous full bypasses; opencode's `--auto` only auto-approves requests that aren't explicitly denied (weaker than `bypassPermissions`) with a separate, more dangerous `--dangerously-skip-permissions` flag explicitly scoped by opencode's own docs to disposable VMs; pi's `-a`/`--approve` is documented only as trust-prompt handling and its coverage of tool-call approval prompts is unconfirmed. Each adapter's choice here is a real security/reliability tradeoff (weaker bypass = more legitimate freezes the watcher must catch; stronger bypass = opencode's own docs caution it for disposable environments only) that the requester should weigh in, not one this spec resolves unilaterally.
4. **"Grok" naming ambiguity (§3)** needs a decision from the requester: xAI's first-party CLI, or the third-party `grok-cli`, before the architect designs the adapter.
5. **Flags drift.** All four CLIs are pre-1.0/actively developed; several flags cited here came from search results and community docs rather than a single authoritative reference for each. The architect/developer phase should re-verify every flag against `<cli> --help` or the shipped binary before finalizing the adapter, per requirement 9.

## 9. Suggested rollout order

Given the findings above, `grok` and `codex` have the clearest, most Claude-Code-like flag parity (an explicit full-bypass flag, and — for grok — an explicit forced-session-id flag), making either a lower-risk first adapter to validate the mechanism end-to-end. `opencode` is a close second (clear model/title/auto-approve flags; session-id-forcing unconfirmed). `pi` is likely the most work per crew member despite its promising `--name` flag, given its permission-bypass semantics are the least confirmed of the four. Recommend building the adapter mechanism itself against **one** CLI first (grok or codex), proving out the descriptor shape and the three-touch-point wiring (`spawn-crew`/`crew-takeover`/`watch-fleet`), then adding the remaining three as independent, smaller follow-ups rather than one combined effort — each is a self-contained descriptor addition once the mechanism exists.
