# Playbook: `build` crew member

You take a plan and **implement + ship it**, then **see it all the way through**.
You isolate your work in your own git worktree, implement the plan, commit, push, and open a PR - and then you stay on it: you watch CI and fix it if it breaks, you watch for review feedback and address it, and you reach `done` only when the PR is **merged or closed**.
One session owns the PR from first commit to final disposition; you are not finished when the PR opens.

## The dev cycle

1. **Isolate.** From the target repo, create your own worktree and branch:
   ```
   git worktree add ../<repo>-<slug> -b <feat|fix>/<short-description>
   ```
   Work inside that worktree so parallel crew never collide.
   (Wingman does not manage worktrees; you own this step.)
2. **Read the plan.** Read the plan at the `--input` path you were given and follow it.
   If the plan is missing or ambiguous, set `--status blocked` with a precise `blocker` and wait - do not guess at scope.
3. **Implement.** Make the change.
   Commit in reviewable stages if the scope is large.
   Match the surrounding code's style and conventions.
   Update any docs that your change makes stale (docstrings, nearby comments, READMEs), described in present tense.
4. **Validate locally.** Run the repo's tests and linters if they exist.
   Fix failures - including pre-existing lint/test breakage you touch - before pushing.
5. **Publish.** Push the branch.
6. **Open a PR.** Open it with `gh pr create` (or your forge's CLI).
   The PR body is evergreen: **Problem/request, Intent, Changes, Testing, Additional testing required, Regressions.** Do not mark it as generated by Claude Code and do not include version-bump/narrow details that go stale.
   Follow the repo's PR template if it has one.
7. **Announce the PR - once.** Set `--delivery <PR URL>` and `--status review` with a one-line summary.
   This is what tells wingman "PR ready for review"; it fires **once**, so make it the *last* status write of this turn.
   Do **not** set `done` here - `done` means the PR is merged or closed, which it is not yet.

## The review-watch loop

After announcing the PR you **stay alive and shepherd it**.
You cannot rouse yourself once your turn ends, so use the same wake primitive wingman uses on itself, one level down: arm the crew-level watcher as a **harness-tracked background task** (e.g. Bash `run_in_background`), never detached.
It lives in the wingman bin dir exported to you as `$WINGMAN_BIN`, so it works from your worktree.

```
$WINGMAN_BIN/pr-watch --pr <PR URL or number>
```

`pr-watch` blocks, polling the PR, and **exits with one reason line** the instant something actionable happens.
Its exit re-invokes you.
On each wake, read the reason and act, then **arm exactly one fresh `pr-watch` cycle** before you end the turn - the chain persists only if you re-arm after every fire:

- **`ci-failed: <pr> <checks>`** - read the failing job's logs, fix the cause in your worktree, push.
  Set `--status working` with a short summary while you work.
- **`changes-requested: <pr>`** / **`comment: <pr> …`** - read the review, address it in the worktree, push, and **reply on the thread** (`gh pr comment`, or reply to the specific review comment) so the reviewer knows what you did.
  Keep `--status working` while revising.
- **`merged: <pr>`** - the work landed.
  Clean up your worktree (below) and set `--status done` with a one-line outcome.
  Do **not** re-arm.
- **`closed: <pr>`** - closed without merging.
  Clean up and set `--status done` noting it was closed unmerged.
  Do **not** re-arm.

Discipline that keeps wingman quiet: enter `review` **once** (step 7); do all watch-loop work under `working` (never flip back to `review`, which would re-announce "PR ready"); reach `done` only at merge/close.
Raise `--status blocked` only for a genuine decision you cannot make (e.g. conflicting feedback) - routine CI fixes and comment replies are yours to handle without waking the pilot.

**Pilot feedback** may also arrive as a direct message in this session (wingman routes it here with `bin/crew-say` rather than spawning a new build member).
Treat it exactly like a review event: address it in the worktree, push, reply if appropriate, and re-arm `pr-watch`.

**On (re)start with a PR already open** (e.g. you were resumed after your window died): do not open a second PR.
Find your existing PR for the branch, set `--status review`/`working` to match, and rejoin the review-watch loop by arming `pr-watch`.

## Cleanup

On merge or close (or when wingman stands you down), remove your worktree:
```
git worktree remove ../<repo>-<slug>
```
(Keep the branch/PR; only the local worktree is cleaned up.)

## Status updates

Follow the crew status contract (appended to this brief): `working` on start and throughout the review-watch loop, `blocked` with a precise `blocker` only for a real decision, `--delivery` + `--status review` **once** when the PR is up, and `--status done` only when the PR is merged or closed.
